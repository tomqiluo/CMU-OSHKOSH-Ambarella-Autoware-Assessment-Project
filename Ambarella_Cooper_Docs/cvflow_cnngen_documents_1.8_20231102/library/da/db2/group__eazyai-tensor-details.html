◆ ea_tensor_data_for_read()

AMBA_API void* ea_tensor_data_for_read ( ea_tensor_t *  tensor,
ea_device_t  device 
)

Get the virtual address of a tensor's memory for read.

When CPU cache is enabled on the tensor's memory, ea_tensor_data_for_*() and ::ea_tensor_phys_for_*() are used together to connect the data flow among EA_VP, EA_DSP and EA_CPU, by cleaning and invalidating the CPU cache.

When using ea_tensor_data_for_*() or ::ea_tensor_phys_for_*(), it's only necessary to know either which device will write the data or which device will read the data, in the calling non-EazyAI API, or the APP source code after. The clean/invalidate operation on the CPU cache will be done automatically inside the call.

The implementation of EazyAI uses ea_tensor_data_for_*() and ::ea_tensor_phys_for_*() inside to connect the data flow, so the data flow can be connected automatically without using ea_tensor_data_for_*() or ::ea_tensor_phys_for_*() in APP, if only using EazyAI API to operate the tensor's memory. Also if ea_tensor_data_for_*(), ::ea_tensor_phys_for_*() and other EazyAI API are used together, it's not necessary to know which device reads/writes the data inside the EazyAI API.

When using ea_tensor_data_for_*() and ::ea_tensor_phys_for_*(), there's no need to use ea_tensor_data(), ea_tensor_phys() and ea_tensor_sync_cache(). But it's OK to use them together if how the data flows is known clearly.

This API is not thread safe when ea_tensor_data_for_*() or ::ea_tensor_phys_for_*() is called in another thread.

Parameters
tensorA pointer which represents a tensor.
deviceA type in ea_device_e which represents the device that will read the data.
Returns
void * The virtual address of the tensor.
Examples
test_api_eazyai_core.cpp.

Create a tensor to store YUV NV12 data with buffer allocated from IAV anonymous memory partition.

Simulator platform: API not supported.

The data type will be EA_U8. The tensor data is stored in the IAV anonymous memory partition so that it can be shared among CPU, DSP and VP. Y data is in the buffer of the created tensor with shape of shape, UV data is in the buffer of ea_tensor_related(). The tensor created with this way can be used to call ea_crop_resize() with device = EA_DSP

AMBA_API void ea_tensor_set_align ( size_t  align)

Set default memory alignment used to create a tensor.

When creating a tensor in NxCxHxW shape, the padding will be added on each W to make the memory size of the data on each W be multiplies of align bytes. The memory size of each W with padding is called pitch, which is equal or greater than the actual size of the valid data on each W.

Simulator platform: padding length is 0, which means that pitch equals to W * element byte size.

Parameters
alignThe alignment of memory in bytes.

◆ ea_tensor_sync_cache()

AMBA_API int ea_tensor_sync_cache ( ea_tensor_t *  tensor,
ea_device_t  from,
ea_device_t  to 
)

Sync cache for tensor's memory (clean or invalidate CPU cache).

When CPU cache is enabled on the tensor's memory, ea_tensor_data(), ea_tensor_phys() and ea_tensor_sync_cache() are used together to connect the data flow among EA_VP, EA_DSP and and EA_CPU, by cleaning and invalidating the CPU cache.

When using ea_tensor_sync_cache(), it's necessary to know which device has written the data and which device will read the data in the called and calling API, or the APP source code before and after. The clean/invalidate operation on the CPU cache will be done automatically inside the call.

When using ea_tensor_data(), ea_tensor_phys() and ea_tensor_sync_cache(), there's no need to use ea_tensor_data_for_*() and ea_tensor_phys_for_*(). But it's OK to use them together if how the data flows is known clearly.

Simulator platform: giving warning, since it's of no use on simulator.

Call this function before reading (or after writing) data on the buffer returned by ea_tensor_data() or ea_tensor_phys(). ea_tensor_sync_cache() is not thread safe, a critical section should be used to protect the data access if there may be conflict in multi threads.

Parameters
tensorA pointer which represents a tensor.
fromA type in ea_device_e which represents the device that has written data.
toA type in ea_device_e which represents the device that will read data.
Returns
int EA_SUCCESS: OK, EA_FAIL: Error
Examples
case_ea_net.cpp, case_ea_preprocess.cpp, case_ea_tensor.cpp, and test_eazyai.c.
