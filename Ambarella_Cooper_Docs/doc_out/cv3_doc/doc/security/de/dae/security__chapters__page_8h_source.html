1300 * @section selinux 2. Secure Enhanced Linux (SELinux)
1301 * @subsection selinux_overview 2.1 SELinux Overview
1302 * @subsubsection selinux_introduction 2.1.1 Introduction
1303 * SELinux is the primary mandatory access control (MAC) mechanism built into a number of GNU / Linux distributions. SELinux originally began as the Flux Advanced Security Kernel
1304 * (FLASK), developed by the Utah University Flux team and the U.S. Department of Defense. The development was later enhanced by the National Security Agency (NSA) and released
1305 * as an open source software (OSS).
1306 *
1307 * @subsubsection core_selinux_components 2.1.2 Core SELinux Components
1308 * The following figure shows a high-level diagram of the SELinux core components that manage the enforcement of the policy and is comprised of the following:
1309 * - An object manager that knows the actions required for a particular resource (such as a file) and can enforce those actions (for example, allows it to write to a file, if permitted by the policy)
1310 * - A security server that makes decisions regarding the subject’s right to perform the requested action on the object, based on the existing security policy rules
1311 * - A security policy that describes the rules using the SELinux policy language
1312 * - An access vector cache (AVC) that improves system performance by caching security server decisions
1313 * @anchor High_Level_Core_SELinux_Components
1314 * @image html high_level_core_selinux_components.png "Figure 2-1. High-Level Core SELinux Components."
1315 *
1316 * @subsubsection mandatory_access_control 2.1.3 Mandatory Access Control
1317 * Mandatory access control is a type of access control in which the operating system (OS) is used to constrain a user or process (the subject) from accessing or performing an
1318 * operation on an object (such as file, disk, memory, and more).
1319 * Each of the subjects and objects include a set of security attributes that can be interrogated by the OS to check if the requested operation can be performed.
1320 * For SELinux, the following are true:
1321 * - Subjects are processes
1322 * - Objects are system resources, such as files and sockets
1323 * - Security attributes are the security context
1324 * - The security server within the Linux kernel authorizes access using the security policy that describes the rules that must be enforced
1325 *
1326 * @subsubsection selinux_users 2.1.4 SELinux Users
1327 * Users in `GNU / Linux` are generally associated with human users (such as Alice and Bob) or operator / system functions (such as admin). While this can be implemented in
1328 * SELinux, SELinux usernames are generally groups or classes of users. For example, all standard system users can be assigned an SELinux username of `user_u`. Administration
1329 * staff can be assigned under `staff_u`.
1330 *
1331 * @subsubsection rbac 2.1.5 Role-Based Access Control
1332 * To further control access to type enforcement (TE) domains, SELinux uses role-based access control (RBAC). This feature enables SELinux users to be associated with one or more
1333 * roles, where each role is then associated with one or more domain types, as shown in the figure below.
1334 * @anchor Role_Based_Access_Control
1335 * @image html role_based_access_control.png "Figure 2-2. Role-Based Access Control."
1336 *
1337 * @subsubsection security_context 2.1.6 Security Context
1338 * SELinux requires security context to be associated with each process (or subject) and object that is used by the security server to decide if access is permitted, as defined by
1339 * the policy.
1340 * The security context is additionally known as a ‘security label’.
1341 * Within SELinux, the security context is represented as variable-length strings that define the SELinux user, their role, a type identifier, and an optional MCS / MLS security
1342 * range level shown below.
1343 * `user:role:type[:range]`
1344 * The following table describes the user, role, type, and range within the security context.
1345 * <table>
1346 * <caption id="security_context">Table 2-1. Security Context.</caption>
1347 * <tr><th align="left">User<th align="left">Role<th align="left">Type<th align="left">Range
1348 * <tr><td>The SELinux user identity can be associated with one or more roles that the SELinux user is allowed to use
1349 * <td>The SELinux role can be associated with one or more types that the SELinux user is allowed to access
1350 * <td>When a type is associated with a process, it defines what processes (or domains) the SELinux user (the subject) can access. When a type is associated with an object, it defines what access permissions are available to the SELinux object
1351 * <td>This field is known as a level; it is only present if the policy supports MCS or MLS. The entry can consist of the following:<br>
1352 * - A single security level that contains a sensitivity level and zero or more categories (for example, s0, s1:c0)
1353 * - A range that consists of two security levels separated by a hyphen (for example, ss0-s15:c0.c1023)
1354 * </table>
1355 *
1356 * @subsubsection selinux_subjects 2.1.7 Subjects
1357 * A subject is an active entity (generally in the form of a person, process, or device) that enables information to flow among objects or that changes the system state.
1358 * Within SELinux, a subject is an active process with an associated security context. However, a process can also be referred to as an object, depending on the context. For example:
1359 * - A running process (such as an active entity) is a subject because it causes information to flow among objects or can change the system state
1360 * - The process can also be referred to as an object because each process has an associated object class called “process.” This process “object” defines what permissions the policy is allowed to grant or deny on the active process
1361 *
1362 * @subsubsection selinux_objects 2.1.8 Objects
1363 * Within SELinux, an object is a resource (such as files, sockets, pipes, or network interfaces) that are accessed via processes (also known as subjects). These objects are
1364 * classified according to which resource they provide access permissions relevant to their purpose (for example, read, receive, and write), and are assigned a security context.
1365 * Each object consists of a class identifier that defines its purpose (for example, file or socket), along with a set of permissions (also known as access vectors) that describe
1366 * what services the object can manage (read, write, send, and so on). When an object is instantiated, it will be allocated a name (for example, a file could be called `configured`,
1367 * a `socket`, or `my_connection`) and a security context (for example, `system_u:object_r:selinux_config_t`).
1368 *
1369 * @subsubsection selinux_modes 2.1.9 SELinux Modes
1370 * SELinux includes three major operation modes:
1371 * - Enforcing: SELinux is enforcing the loaded policy
1372 * - Permissive: SELinux has loaded the policy, but is not enforcing the policy rules. This is generally used for testing as the audit log contains the AVC-denied messages
1373 * - Disabled: The SELinux infrastructure is not enabled; therefore, no policy can be loaded
1374 * These flags are set in the `/etc/selinx/config` file.
1375 *
1376 * @subsubsection auditing_selinux_events 2.1.10 Auditing SELinux Events
1377 * SELinux includes two main types of audit events: AVC audit events and SELinux-aware application events.
1378 * AVC audit events are generated by the AVC subsystem as a result of access denials, or where specific events have requested an audit message.
1379 * SELinux-aware application events are generated by the SELinux kernel services and SELinux-aware applications for events, such as system errors, initialization, policy loading,
1380 * changing Boolean states, setting of enforcing / permissive mode, and relabeling.

1509 * `sestatus` is used to obtain the status of the system running SELinux.
1510 * - Usage: `sestatus [OPTION]`
1511 * - `-v`: verbose check of process and file contexts

1530 * `getenforce` is used to get the mode SELinux is running in. `setenforce` is used to modify the mode SELinux is running in.
1531 * Use `Enforcing` or `1` to put SELinux in enforcing mode. Use `Permissive` or `0` to put SELinux in permissive mode.
1532 * For example:
1533 * @code

1536 * board# setenforce 1
1537 * board# getenforce

1539 * @endcode
1540 * @note

1542 * For more information, refer to @ref selinux_config_files.

1556 * -r ROLE
1557 * -t TYPE
1558 * -l RNG

1591 * @subsubsection policy_config_files 2.2.5 Policy Configuration Files
1592 * Each file discussed in this section is related to the policy name, as follows:
1593 * `/etc/selinx/<policy_name>` (Ambarella uses `targeted` by default)
1594 * - `Seusers`<br>
1595 * The seusers file is used by login programs and maps the GNU / Linux user to the SELinux user. A typical login sequence would be as follows:
1596 * - Using the GNU / Linux `user_id`, look up the `seuser_id` from this file. If an entry cannot be found, use `__default__entry`.
1597 * - To determine the remaining context to be used as the security context, read the `./contexts/users/[user_id]` file. If this file is not present, check for a default context in the `./contexts/default_contexts` file. If no default context is found, read the `./contexs/failsafe_context` file to enable a failed safe context to be set.
1598 * Example of seusers file contents:
1599 * @code
1600 * system_u:system_u
1601 * root:root
1602 * __default__:user_u
1603 * @endcode
1604 *
1605 * - `Policy / policy.<ver>`<br>
1606 * This is the binary policy file that is loaded into the kernel to enforce the policy.
1607 * - `Contexts / customizable_types`<br>
1608 * The `customizable_types` file contains a list of types that will not be re-labeled by the `setfiles` or `restorecon` commands. The commands check this file before re-labeling and exclude those in the list unless the `–F` flag is used.
1609 * Example of file contents:
1610 * @code
1611 * container_file_t
1612 * sandbox_file_t
1613 * svirt_image_t

1616 * virt_content_t
1617 * httpd_user_htaccess_t

1623 * home_bin_t
1624 * user_tty_device_t
1625 * @endcode
1626 *
1627 * - `Contexts / default_contexts`<br>
1628 * The `default_context` file is used by SELinux-aware applications that are required to set a security context for user processes (generally the login applications).
1629 * The following is an example of `default_contexts` contents:
1630 * @code
1631 * system_r:crond_t:s0 system_r:system_crond_t:s0

1673 * @subsubsection customize_policy 2.2.6 Customize Policy
1674 * Users can customize their own policy to fit their systems by adding a new SELinux module.
1675 * <b>Create Module CIL Files</b><br>
1676 * To create a new module, create a `cil` file and write the rules:
1677 * @code
1678 * board# vi test_policy.cil
1679 * (allow systemd_logind_t init_var_lib_t (lnk_file (read)))
1680 * @endcode
1681 * This rule indicates that when the source file is `system_logind_t`, the targeted file is `init_var_lib_t`, the object class is `lnk_file`, and the object permission is `read`.
1682 * Thus, the operation is allowed.
1683 *
1684 * <b>Install the Module</b><br>
1685 * Use the semodule `–i` to install the module:
1686 * @code
1687 * board# semodule –i test_policy.cil
1688 * @endcode
1689 *
1690 * <b>Update the Module</b><br>
1691 * -# Obtain `cil` files from the module (if the user already has the cil files, this step can be skipped).
1692 * @code
1693 * board# semodule -c --extract test_policy
1694 * @endcode
1695 * -# Add the rules to the `cil` file.
1696 * @code
1697 * board# echo '(allow systemd_rfkill_t init_var_lib_t (lnk_file (read)))' >> test_policy.cil
1698 * @endcode
1699 * -# Install a new module.
1700 * @code
1701 * board#semodule –i test_policy.cil
1702 * @endcode
1703 *
1704 * <b>Check the Policy</b><br>
1705 * Use `selinux_check_access` to determine if the rules have been written into the policy.
1706 * @code
1707 * board# selinux_check_access system_u:system_r:systemd_rfkill_t:s0 system_u:object_r:init_var_lib_t:s0 lnk_file read
1708 * @endcode
1709 * If the module was not successfully installed, the following error message appears:
1710 * @code
1711 * avc: denied { read } for scontext=system_u:system_r:systemd_rfkill_t:s0 tcontext=system_u:object_r:iit_var_lib_t:s0 tclass=lnk_file permissive=1
1712 * @endcode

1714 * @section tee_and_trustzone 3. TEE and Arm TrustZone
1715 * TEE is a specification / standard from GlobalPlatform (GP), a non-profit industry association. One of TEE’s implementations is Arm TrustZone.
1716 * For more information about GP, visit the following link: @ref https://globalplatform.org/.
1717 * TEE specification can be found here: @ref https://globalplatform.org/specs-library/?filter-committee=tee
1718 * For more information about TEE programming, users can refer to TEE API and TEE client API specifications as follows: <em>GPD_TEE_Internal_Core_API_Specification_v1.3.1_PublicRelease_CC.pdf</em> and <em>TEE_Client_API_Specification-V1.0_c.pdf</em>
1719 *
1720 * @subsection arm_trustzone 3.1. Arm® TrustZone®
1721 * Arm TrustZone technology offers an efficient, system-wide approach to security with hardware-enforced isolation built into the CPU. TrustZone provides the perfect starting
1722 * point for establishing a device root of trust (RoT) based on platform security architecture (PSA) guidelines.
1723 * For more information, visit @ref https://www.arm.com/technologies/trustzone-for-cortex-a.
1724 * @anchor The_System_Diagram_of_Arm_TrustZone
1725 * @image html sys_diag_of_arm_trustzone.png "Figure 3.1. System Diagram of Arm TrustZone"
1726 * On Ambarella platforms, Arm TrustZone can be enabled in the flexible Linux SDK.
1727 * In the flexible Linux SDK, Linux functions as the rich OS, the open portable trusted execution environment (OP-TEE) as the trusted operating system (TOS), and Arm trusted firmware
1728 * (ATF) as the trusted firmware / secure monitor. The hypervisor and multiple-rich OS are not implemented in the SDK.
1729 * The following diagram illustrates Arm TrustZone in the Ambarella Flexible Linux SDK.
1730 * @anchor Arm_TrustZone_in_Linux_SDK
1731 * @image html arm_trustzone_in_linux_sdk.png "Figure 3.2 Arm TrustZone in Linux SDK"
1732 *
1733 * @subsection arm_trusted_firmware 3.2. Arm Trusted Firmware
1734 * ATF is the reference software for Arm TrustZone. It implements a secure monitor, which manages the switch between the normal world (rich OS) and the secure world (secure OS).
1735 * ATF also implements authentication during the boot stage (secure boot). During this process, the Ambarella Linux SDK employs a partial authentication chain for secure boot via
1736 * the built-in hardware engine: BL2 authenticates BL31, BL32, and BL33.
1737 *
1738 * @subsection op_tee 3.3. OP-TEE
1739 * OP-TEE is an open source project, which implements the GlobalPlatform TEE system architecture specifications and serves as a TOS. OP-TEE provides the TEE application programming
1740 * interface (API) and the TEE client API, and implements a TEE-suppliant. The TEE API, as defined by the GlobalPlatform TEE standard, is for the development of trusted applications
1741 * (TAs). The TOS is accessible from the rich OS (Linux) using the GlobalPlatform TEE client API specification. The TEE client API is used to trigger the secure execution of
1742 * applications within the TEE. The TEE-supplicant loads the TA from the rich OS into the secure OS for execution.
1743 * The following diagram shows the software stack of Arm TrustZone (Linux, OP-TEE, and ATF).
1744 *
1745 * @anchor The_Software_Stack_of_a_TrustZone
1746 * @image html sw_stack_of_trustzone.png "Figure 3.3 Software Stack of TrustZone (Linux, OP-TEE, and ATF)"
1747 *
1748 * @note
1749 * - Refer to @ref https://www.op-tee.org/ to access the OP-TEE home page.<br>
1750 * - Refer to @ref https://github.com/OP-TEE/ to access the project.<br>
1751 * - Refer to @ref https://optee.readthedocs.io/ to check OP-TEE documents.

1753 * @subsection ta_and_ca 3.4. Trusted Application and Client Application
1754 * The TA functions in the secure world in OP-TEE. It can be a user TA, which operates in the TEE user space, or a pseudo TA (PTA), which operates in the TEE kernel space.
1755 * The user TA includes a signature and resides in the normal world storage. The PTA is embedded in the trusted OS (OP-TEE-OS). Each TA includes a universally unique identifier
1756 * (UUID) as its identity. When a normal TA is requested, it is loaded by a TEE-supplicant in rich the OS and conveyed to the secure world after the digital signature has been
1757 * verified.
1758 * The client application (CA) functions in the normal world.
1759 * The SDK locations for the modules are as follows:
1760 * - TEE_Client_API: `SDK/ambarella/boot/tee/optee_client/libteec`
1761 * - RPC service: `SDK/ambarella/boot/tee/optee_client/tee-supplicant`
1762 * - OP-TEE driver: `SDK/ambarella/kernel/linux-5.15/drivers/tee/optee`
1763 * - TEE_Internal_API: `SDK/ambarella/boot/tee/optee_os/lib/libutee/include`
1764 * OP-TEE’s user TAs are signed by `SDK/ambarella/boot/tee/optee_os/keys/default_ta.pem`.
1765 * @note
1766 * Each TA in the secure world should include a UUID to enable the CA application to locate it through the TEE API in the normal world.

1777 * @section anti_rollback 4. Anti-Rollback
1778 * To prevent the software version from being rolled backed by a hacker (a method used by hackers to exploit known security vulnerabilities in older software versions), the
1779 * Ambarella CV5x / CV7x series provides a hardware monotonic counter, which can be used for anti-rollback.
1780 *
1781 * @section disable_hw_debug_port 5. Disable the Hardware Debug Port
1782 * Ambarella recommends removing or disabling hardware debug ports, including force USB boot and joint test action group (JTAG).
1783 * On CV5x, force USB boot is disabled when secure boot is enabled. Additionally, there is an eFUSE bit that can be used to disable JTAG.
1784 *
1785 * @section disable_no_secure_protocol 6. Disable the Non-Secure Protocol / Port
1786 * Ambarella recommends disabling all ports for non-secure protocols. The following is a widely used, non-secure protocol list: Telnet (port 23), file transfer protocol (FTP)
1787 * (port 21), trivial file transfer protocol (TFTP) (port 69), hypertext transfer protocol (HTTP) (port 80), and system gateway (port 19531).
1788 * For remote login, secure shell (SSH) is more secure than Telnet and HTTPS is more secure than HTTP.
1789 * The following command enables users to check what port is opened using the portscan tool, `nmap`:
1790 * @code
1791 * #apt-get install nmap
1792 * nmap -A 10.0.0.2
1793 * nmap -A –p1-65535 10.0.0.2
1794 * @endcode

1796 * @section buffer_overflow_attack_resistance 7. Buffer Overflow Attack Resistance
1797 * Hackers commonly use buffer overflow attacks. To resist a buffer overflow attack, Ambarella recommends that users enable the following items supported by the SDK.
1798 * Additionally, users should remove / reduce software bugs, especially string manipulation functions. Users are forbidden from using unsafe string functions, such as `strcpy`,
1799 * `strcat`, and `sprintf`. As an alternative, Ambarella recommends using `strncpy` and `snprintf`.
1800 *
1801 * @subsection add_space_layout_randomization 7.1. Address Space Layout Randomization
1802 * Address space layout randomization (ASLR) increases the difficulty of a buffer overflow attack. Linux supports ASLR. Use the command below to enable it on Ambarella chips:
1803 * @code
1804 * echo 2 > /proc/sys/kernel/randomize_va_space
1805 * @endcode
1806 *
1807 * @subsection pos_independent_executable 7.2. Position Independent Executable
1808 * Position independent executable (PIE) enables code to be executed at different places, thus increasing the difficulty of a buffer overflow attack. PIE is supported by the GNU
1809 * compiler collection (GCC), as shown below:
1810 * @code
1811 * -fPIE -pie
1812 * @endcode
1813 *
1814 * @subsection no_execute_protect 7.3. No-eXecute Protect
1815 * No-eXecute protect (NX) prevents execution on the no-execute part and is enabled, by default, in GCC. The following demonstrates how to enable and disable NX.
1816 * - Disable NX:
1817 * `-z execstack`
1818 * - Enable NX:
1819 * `-z noexecstack`
1820 *
1821 * @subsection stack_protector 7.4. Stack Protector
1822 * GCC supports stack protection. The compile option enables the stack protector.
1823 * - Protect functions that have string / char local variables:
1824 * `-fstack-protector`
1825 * - Protect all functions:
1826 * `-fstack-protector-all`
1827 *
1828 * @subsection fortify 7.5. FORTIFY
1829 * FORTIFY is a GCC feature that can be used for code and runtime checks if there is a potential buffer overflow risk.
1830 * - Buffer overflow check at the compile stage:
1831 * `-D_FORTIFY_SOURCE=1`
1832 * - Buffer overflow check at both the compile and runtime stages:
1833 * `-D_FORTIFY_SOURCE=2`
1834 *
1835 * @subsection read_only_relocation 7.6. Read-Only Relocation
1836 * GCC supports read-only relocation (RELRO), which means the relocation region is read-only. RELRO can increase the difficulty of a buffer overflow attack and can be used to
1837 * protect the global offset table (GOT).
1838 * By default, the RELRO setting is partial (partial RELRO means the GOT is writable).
1839 * - Disable RELRO:
1840 * `-z norelro`
1841 * - Partial RELRO:
1842 * `-z,relro -z lazy`
1843 * - Full RELRO:
1844 * `-z,relro -z,now`
1845 *
1846 * @section dram_scrambling 8. DRAM Scrambling
1847 * Dynamic random-access memory (DRAM) scrambling is automatically enabled when secure boot is enabled; it is designed to prevent a potential sniffer on the physical DRAM memory.
1848 *
1849 * @section cve_fix_strategy 9. CVE Fix Strategy
1850 * Regarding CVE fixes, Ambarella synchronizes up to the latest version of the Linux kernel and open sourse software (OSS) modules, then provide the updates inside the SDK package
1851 * (quarterly or bi-quarterly) for both Linux kernel and OSS module fixes.
1852 * For these open source modules, Ambarella relies only on the community updates, but will not provide its own fixes for these modules.
1853 * This is a general support rule of CVE fixes for all Cooper Linux SDK customers.
